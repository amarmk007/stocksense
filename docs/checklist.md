## Checklist

- [x] **1. Project scaffold**
  Spec ref: `spec.md > File Structure`
  What to build: Create the full repo structure from the spec's file tree. Initialize the ASP.NET Core Web API project (`StockSense.API`) with .NET 9. Initialize the React + Vite + TypeScript frontend. Install all dependencies: Tailwind CSS, Framer Motion, Axios for frontend; EF Core 9, Hangfire, Identity, JWT Bearer, Google OAuth NuGet packages for backend. Add a root `README.md`. Wire up `vite.config.ts` with a proxy so `/api` calls hit the local backend during development. No features — just a compiling, running scaffold with placeholder routes.
  Acceptance: `dotnet run` starts the API on localhost with no errors. `npm run dev` starts the React app with no errors. Navigating to the frontend shows a blank page with no console errors.
  Verify: Run both servers and confirm both start without errors. Open the browser to the Vite dev URL and confirm no console errors.

- [x] **2. Data model + EF Core setup**
  Spec ref: `spec.md > Data Model`
  What to build: Create all four EF Core model classes: `User`, `UserProfile`, `RecommendationSet`, `RecommendationItem` with all columns from the spec. Create `AppDbContext` with all four `DbSet<>` properties and relationship configuration. Configure PostgreSQL connection via `DATABASE_URL` env var in `Program.cs`. Run `dotnet ef migrations add InitialCreate` and `dotnet ef database update`. Seed no data.
  Acceptance: All four tables exist in PostgreSQL with correct columns and foreign keys. `dotnet ef migrations list` shows `InitialCreate` as applied.
  Verify: Connect to the PostgreSQL database (Railway or local) and confirm the four tables exist with correct schema. Run `dotnet ef migrations list` and confirm the migration shows as applied.

- [x] **3. Authentication — Google OAuth + JWT**
  Spec ref: `spec.md > Authentication`
  What to build: Backend — `AuthController` with `GET /api/auth/google` (redirect to Google) and `GET /api/auth/google/callback` (upsert user, issue JWT with `IsOnboarded` claim, redirect to React with `#token=...`). Configure `Microsoft.AspNetCore.Authentication.Google` + `JwtBearer` in `Program.cs`. Apply `[Authorize]` globally; mark auth endpoints `[AllowAnonymous]`. Frontend — `Auth.tsx` page (sign-in button → redirect to `/api/auth/google`). `useAuth.ts` hook: extract JWT from URL fragment on callback, store in memory only (React state/context, never localStorage). `App.tsx` route guards: unauthenticated → redirect to Auth; authenticated + not onboarded → redirect to onboarding; authenticated + onboarded → dashboard. Silent re-auth: on page load, if no token in memory, redirect to Google OAuth.
  Acceptance: Clicking "Sign in with Google" completes the OAuth flow and lands the user on the onboarding route. Refreshing the page triggers silent re-auth if the user has an active Google session. A request to `/api/profile` without a JWT returns 401.
  Verify: Complete the Google OAuth flow end-to-end. Confirm the JWT lands in memory (check React DevTools or add a console.log in useAuth). Confirm a direct API call without the token returns 401. Refresh the page and confirm silent re-auth redirects to Google and comes back without a manual login.

- [ ] **4. Onboarding flow**
  Spec ref: `spec.md > Onboarding`
  What to build: Frontend — `OnboardingFlow.tsx` with 4 screens in sequence (TikTok-style slide transition using Framer Motion `AnimatePresence`): (1) investment amount slider $1k–$1M in $10k steps, (2) timeline slider 1–10 years in 1-year steps, (3) expected return slider 5%–100% in 5% steps, (4) experience level picklist (Novice / Experienced). "Get Recommendations" CTA enabled only after all 4 inputs are set. On CTA tap: `POST /api/profile` with all 4 values. After POST succeeds, start polling `GET /api/recommendations/status` every 3 seconds. Show loading screen with message "Analyzing markets for you..." during polling. When status returns `"ready"`, navigate to dashboard. Backend — `ProfileController`: `POST /api/profile` saves `UserProfile`, sets `IsOnboarded = true` on the `User` record, enqueues a Hangfire background job immediately. `GET /api/recommendations/status` returns `{ status: "pending" | "ready" }` — "ready" when a non-stale `RecommendationSet` exists for the user. Route guard: if `IsOnboarded = true` in JWT claims, skip onboarding and go to dashboard.
  Acceptance: New user completes all 4 onboarding screens, taps CTA, sees loading screen, then lands on dashboard once recommendations are ready. A returning user (IsOnboarded = true) skips onboarding entirely.
  Verify: Complete onboarding as a new user — confirm each screen transitions correctly, CTA is disabled until all 4 answered, loading screen appears after POST. Sign out and back in as the same user — confirm onboarding is skipped and dashboard loads directly.

- [ ] **5. External data services — Finnhub + SEC EDGAR**
  Spec ref: `spec.md > Recommendation Engine > FinnhubService` and `spec.md > Recommendation Engine > EdgarService`
  What to build: `FinnhubService.cs` — three calls per ticker: `GET /quote`, `GET /company-news`, `GET /stock/recommendation`. Auth via `?token={FINNHUB_API_KEY}` query param. Base URL: `https://finnhub.io/api/v1`. Return typed C# objects (quote price, news list, analyst ratings list). `EdgarService.cs` — one call per ticker: `GET https://efts.sec.gov/LATEST/search-index?q="{ticker}"&dateRange=custom&startdt={30 days ago}&enddt={today}&forms=8-K,10-K`. No API key. Extract filing title + URL from response. Register both services in DI in `Program.cs`. Add `FINNHUB_API_KEY` to `appsettings.Development.json` (local dev only — never commit secrets).
  Acceptance: Both services can be called for a test ticker (e.g., "AAPL") and return valid data. Finnhub returns a current quote price, recent news items, and analyst ratings. EDGAR returns at least one recent filing with a title and URL.
  Verify: Add a temporary test endpoint or unit test that calls both services with "AAPL" and logs the response. Confirm Finnhub returns quote data and EDGAR returns at least one filing. Remove the test endpoint before the next step.

- [ ] **6. ClaudeService + DailyRecommendationJob**
  Spec ref: `spec.md > Recommendation Engine > ClaudeService` and `spec.md > Recommendation Engine > Daily Hangfire Job`
  What to build: `ClaudeService.cs` — two-pass Claude call using the Anthropic C# SDK (`Anthropic` v12.11.0). Pass 1: system prompt as stock analyst, user message with profile data, response is a JSON array of 5-8 tickers. Add system prompt note: "Only recommend US-listed stocks available on NYSE or NASDAQ." Pass 2: inject Finnhub + EDGAR data into the prompt; use Claude tool use with the `submit_recommendations` tool schema from the spec to enforce structured JSON output. Experience-level prompt injection: Novice gets plain-English guidance, Experienced gets data-density instruction (exact text from spec). Model: `claude-sonnet-4-6`. `DailyRecommendationJob.cs` — Hangfire job that iterates all users sequentially. For each user: call ClaudeService Pass 1 (get tickers), call FinnhubService for each ticker, call EdgarService for each ticker, call ClaudeService Pass 2 (full recommendations), save `RecommendationSet` + `RecommendationItems` to DB, mark previous set `IsStale = true`. Add 1000ms delay between users. Configure Hangfire in `Program.cs` with PostgreSQL storage and cron `"0 5 * * *"`. Secure the Hangfire dashboard: require admin role or restrict to localhost only.
  Acceptance: Triggering the job manually (via Hangfire dashboard or test endpoint) for a real user with a saved profile produces a `RecommendationSet` with multiple `RecommendationItems` in the database. Each item has ticker, name, upside estimate, reasoning, signals JSON, and sources JSON populated.
  Verify: Trigger the job manually for your test user. Query the database and confirm a `RecommendationSet` row and 5+ `RecommendationItem` rows exist with all fields populated. Check the Hangfire dashboard to confirm the job ran successfully and no errors were thrown.

- [ ] **7. Recommendation API + Dashboard UI**
  Spec ref: `spec.md > Dashboard` and `spec.md > Recommendation Engine > Stale Data Handling`
  What to build: Backend — `RecommendationsController`: `GET /api/recommendations` returns the latest `RecommendationSet` for the authenticated user in the response shape from the spec (with `isStale`, `generatedAt`, and `recommendations` array). If no set exists, return 404. Frontend — `Dashboard.tsx`: fetches `GET /api/recommendations` on mount, handles 3 states: (1) loading spinner while fetching, (2) stale state — renders cards + `StaleBanner.tsx` with "Today's recommendations are being prepared — showing yesterday's", (3) ready state — renders cards. `RecommendationCard.tsx`: collapsed state shows ticker, company name, upside estimate badge. Expanded state shows reasoning paragraph, signals in 3 typed sections (analyst / macro / market), sources list. Expand/collapse via Framer Motion `AnimatePresence`. No frontend branching for experience level — text from Claude handles it. `StaleBanner.tsx`: simple banner component shown when `isStale: true`.
  Acceptance: Dashboard renders 5+ recommendation cards. Clicking a card expands it smoothly and shows reasoning, signals (3 sections), and sources. Clicking again collapses it. A stale set shows the banner. Error state shows a non-technical error message.
  Verify: Run the app, sign in, complete onboarding (or use an existing user with recommendations). Confirm 5+ cards render on the dashboard. Expand and collapse 2-3 cards — confirm animations work and all content sections appear. Manually set `IsStale = true` in the DB and refresh — confirm the banner appears.

  Verify: Navigate to the profile page from the dashboard. Change the investment amount and save — confirm the confirmation message appears. Refresh the page — confirm the new value is still set. Go back to the dashboard — confirm recommendations are unchanged (no re-generation triggered).
